-- Ejercicio nro. 5:  

Escenario A (por defecto: READ COMMITTED)
user_medico:

sql
SELECT * FROM patologia WHERE id_patologia = 1;
→ Ve los datos actuales de la patología 1.

user_postgres:

sql
BEGIN;  -- sin indicar aislamiento, usa READ COMMITTED
UPDATE patologia SET nombre = 'Apellido' WHERE id_patologia = 1;
COMMIT;
user_medico vuelve a consultar:

sql
SELECT * FROM patologia WHERE id_patologia = 1;
→ Ahora ve el nombre actualizado (con el apellido).

Nivel de aislamiento: READ COMMITTED (por defecto).
Anomalía: Lectura no repetible (non-repeatable read). El mismo SELECT devuelve resultados distintos porque otro usuario confirmó cambios en el medio.
Cómo evitarlo: Usar REPEATABLE READ o SERIALIZABLE para garantizar consistencia en las lecturas dentro de la transacción.

--> ECENARIO B (REPEATABLE READ)
user_medico:

sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT * FROM patologia WHERE id_patologia = 1;
→ Ve los datos iniciales de la patología 1.

user_postgres:

sql
BEGIN;  -- por defecto READ COMMITTED
UPDATE patologia SET estado = 'Nombre' WHERE id_patologia = 1;
COMMIT;
user_medico vuelve a consultar dentro de su transacción:

sql
SELECT * FROM patologia WHERE id_patologia = 1;
→ Sigue viendo los datos iniciales, porque en REPEATABLE READ todas las lecturas son consistentes con el estado al inicio de la transacción.

user_postgres consulta:

sql
SELECT * FROM patologia WHERE id_patologia = 1;
→ Ve el estado actualizado (con su nombre).

user_medico hace COMMIT y vuelve a consultar:

sql
COMMIT;
SELECT * FROM patologia WHERE id_patologia = 1;
→ Ahora sí ve el estado actualizado.

Nivel de aislamiento: REPEATABLE READ.
Anomalía evitada: No hay lecturas no repetibles dentro de la transacción del médico.
Comportamiento: El médico trabajó con una “foto” consistente de los datos hasta que cerró la transacción.

        RAISE NOTICE 'Rollback aumento 30%, aplicar 20%';
